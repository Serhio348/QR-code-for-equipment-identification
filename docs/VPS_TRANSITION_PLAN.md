# План перехода проекта на VPS (Docker + Nginx) — с Supabase Auth (Cloud)

Цель: развернуть проект на собственном VPS в “микросервисном” стиле (несколько сервисов за одним Nginx reverse proxy), **не ломая текущую модель авторизации Supabase**, и параллельно начать разработку отдельного микросервиса опроса обратного осмоса, чтобы лучше разобраться с Docker и Nginx.

Этот документ описывает практический план **переезда на VPS**. Для отдельной большой миграции “Supabase → полностью на свой сервер” уже есть черновик `docs/MIGRATION_PLAN.md`; здесь мы используем его как справочник и выносим “self-hosted Supabase” в отдельный этап “потом”.

---

## 0) Текущая картина проекта (что именно будем размещать на VPS)

Минимальный набор сервисов проекта сейчас:

- **Frontend (SPA)**: корень репозитория, Vite build → статика, отдаётся Nginx (`nginx.conf`).
- **Backend (AI Consultant API)**: `ai-consultant-api/` — Node/Express API (порт из `PORT`, CORS через `ALLOWED_ORIGINS`).
- **Фоновые задачи (cron)**:
  - сбор показаний Beliot: сейчас запускается через GitHub Actions (`.github/workflows/collect-readings.yml`)
  - синхронизация счетов bvod.by: GitHub Actions (`.github/workflows/invoice-sync.yml`)
- **OSMOS/Modbus**:
  - сейчас есть CLI-скрипт `scripts/read-weintek-osmos-modbus.ts` (ручной запуск)
  - планируется новый **микросервис** (poll каждые 5 сек + REST API)

Хранилище и авторизация:

- **Supabase (Cloud)**: Auth + DB + Storage (по README и коду).
- В приложении используются `VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY`.

## 1) Целевая архитектура на VPS (рекомендуемый вариант)

### 1.1. Поддомены (рекомендуется при логине/ролях)

Используем разные поддомены (меньше конфликтов маршрутизации, проще безопасность):

- `app.<domain>` → SPA (static)
- `api.<domain>` → `ai-consultant-api`
- `osmos.<domain>` → osmos-modbus микросервис

Плюс:
- меньше “магии” с проксированием путей (`/api`, `/osmos`)
- проще включать rate limit / basic-auth / VPN per сервис
- проще мигрировать сервисы отдельно

Минус: нужен корректный **CORS** для `api.<domain>` (это уже предусмотрено в `ai-consultant-api` через `ALLOWED_ORIGINS`).

### 1.2. Docker Compose на сервис

На сервере держим один Nginx (reverse proxy, TLS, routing), а каждый сервис запускаем в Docker (по одному `docker compose` проекту).

Это даёт:
- изоляцию зависимостей
- одинаковый “prod” запуск для всех сервисов
- понятный откат через теги/сборку

## 2) Принципы деплоя (чтобы не усложнять жизнь)

### 2.1. “12-factor” правила для каждого сервиса

Для каждого микросервиса:
- порт только из `PORT`
- `/health` — публичный endpoint (без auth)
- конфигурация только через env vars (никаких “зашитых” адресов)
- логи в stdout/stderr (Docker их собирает)

### 2.2. Секреты

Никогда не хранить в git:
- `SUPABASE_SERVICE_ROLE_KEY`, ключи AI-провайдеров, пароли, `INVOICE_SYNC_SECRET`

На VPS секреты хранить:
- либо в `.env` рядом с compose-файлом в `/opt/services/<service>/.env`
- либо в secret manager (если появится)

### 2.3. Фронтенд: важный момент про Vite env

Сейчас `Dockerfile` фронтенда собирает Vite с `VITE_*` на этапе build (см. существующий `Dockerfile`).
Это означает:
- чтобы поменять API URL в фронте — надо пересобрать образ (это нормально для прод-пайплайна)

Если захочется runtime-конфиг без пересборки — это отдельная доработка (не обязательна на первом этапе).

## 3) План переезда: этапы и порядок (рекомендуемый)

Ниже — план так, чтобы вы могли остановиться на любом этапе и система оставалась работоспособной.

### Этап A — подготовка сервера (инфраструктура)

**Цель**: поднять базовую инфраструктуру, не трогая приложение.

1) Подготовить DNS:
   - `A app.<domain> -> <VPS_IP>`
   - `A api.<domain> -> <VPS_IP>`
   - `A osmos.<domain> -> <VPS_IP>`

2) Установить базовое на Ubuntu:
   - Docker + Docker Compose plugin
   - Nginx
   - UFW (или альтернативный firewall)

3) Настроить firewall:
   - открыть только 22 (SSH), 80/443 (HTTP/HTTPS)
   - сервисные порты контейнеров наружу **не публиковать**, только `127.0.0.1` либо docker network

4) Настроить TLS (Let’s Encrypt) для поддоменов:
   - либо через `certbot --nginx`
   - либо через отдельный контейнер (например, nginx-proxy + acme-companion) — но лучше начать с certbot, проще отладка

Критерий готовности:
- `https://app.<domain>` открывается (пока может показывать заглушку)
- Nginx reload/test проходят

### Этап B — деплой backend `ai-consultant-api` на VPS

**Цель**: запустить API на `api.<domain>`.

1) Создать на VPS директорию:
   - `/opt/services/ai-consultant-api/`

2) Создать `docker-compose.yml` (на VPS) для `ai-consultant-api`:
   - контейнер Node
   - `PORT=3001` (внутренний)
   - `ALLOWED_ORIGINS=https://app.<domain>` (и при необходимости `http://localhost:...` для теста)
   - `SUPABASE_URL` и `SUPABASE_SERVICE_KEY` (service role, только серверу!)

3) Настроить Nginx:
   - `api.<domain>` → проксировать на `localhost:3001`

4) Проверка:
   - `GET https://api.<domain>/health` возвращает OK
   - защищенные эндпоинты отвечают предсказуемо (401/403 без токена)

Критерий готовности:
- API стабильно отвечает после перезапуска сервера (docker restart policy / systemd для docker)

### Этап C — деплой frontend (SPA) на VPS

**Цель**: запустить SPA на `app.<domain>`, которая ходит на `https://api.<domain>`.

1) Подготовить build-параметры Vite для production:
   - `VITE_AI_CONSULTANT_API_URL=https://api.<domain>`
   - `VITE_SUPABASE_URL=...`
   - `VITE_SUPABASE_ANON_KEY=...`
   - другие `VITE_*` по необходимости

2) Собрать docker image и задеплоить контейнер со статикой (nginx в контейнере, как сейчас).

3) Nginx:
   - `app.<domain>` → отдавать SPA (или проксировать на контейнер со статикой)

4) Проверки:
   - вход/регистрация (Supabase Auth Cloud) работает
   - после логина роли/доступы в UI корректны
   - запросы к `api.<domain>` не блокируются CORS

Критерий готовности:
- пользователь может зайти в систему и пользоваться основными экранами

### Этап D — микросервис опроса осмоса (osmos-modbus)

**Цель**: запустить отдельный сервис на `osmos.<domain>` или за API, и убедиться, что он стабильно опрашивает Modbus.

Рекомендуемая архитектура сервиса:
- Node.js + TypeScript
- background poll (каждые 5 секунд) → хранить `lastSnapshot` в памяти
- `GET /api/osmos` → отдаёт последнюю выборку + timestamp + “stale/fresh”
- `GET /health` → состояние (последний успешный poll, ошибка, latency)

Деплой:
- контейнер Node, порт например `3003`
- Nginx: `osmos.<domain>` → `localhost:3003`

Безопасность:
- если сервис внутренний, лучше:
  - либо закрыть доступ (VPN/allowlist IP)
  - либо basic-auth на Nginx
  - либо проверка JWT (Supabase) и role-based доступ (если нужно из браузера)

Критерий готовности:
- `GET /api/osmos` отдаёт данные с интервалом обновления ~5 секунд
- при потере Modbus устройство не “умирает”, а показывает ошибку и продолжает ретраи

### Этап E — перенести cron jobs с GitHub Actions на VPS (опционально)

**Вариант 1 (быстрее и надёжнее для старта): оставить GitHub Actions**
- плюсы: не зависит от uptime VPS, уже работает
- минусы: внешний мир; секреты в GitHub (но это норм)

**Вариант 2: перенести на VPS**
- systemd timer / cron + запуск node-скриптов в контейнере
- либо отдельный “worker” сервис (контейнер), который по расписанию делает задачи

Рекомендация:
- на первом этапе оставить GitHub Actions
- переносить на VPS после стабилизации сервисов

## 4) CI/CD на VPS через GitHub Actions (рекомендуемый минимум)

Цель: “push в main → задеплоилось на VPS”.

Упрощенная схема:
1) GitHub Actions собирает Docker image (или собирает на VPS).
2) По SSH запускает команды на VPS:
   - `git pull` (или получает артефакт)
   - `docker compose up -d --build`
   - health-check

### Варианты сборки

**A) Build на VPS**
- проще (не нужен registry)
- медленнее, но для маленьких проектов нормально

**B) Build в GitHub + push в registry**
- быстрее, чище
- требует registry (GHCR) и аккуратной работы с тегами

Рекомендация на старт: build на VPS, потом перейти на GHCR.

## 5) Набор проверок перед “боевым” переключением

### Функциональные
- логин/логаут
- роли: admin/user доступы
- основные экраны грузятся
- `api.<domain>/health` OK
- cron (если на VPS) выполняется и пишет логи
- osmos сервис даёт данные и переживает потери связи

### Безопасность
- `SUPABASE_SERVICE_ROLE_KEY` нигде не попадает во фронтенд
- Nginx ограничивает методы/размеры запросов на чувствительных эндпоинтах
- закрыты лишние порты

### Операционка
- бэкап стратегии (если база остаётся в Supabase Cloud — это их зона ответственности, но ваши секреты и конфиги — ваши)
- логирование контейнеров и ротация

## 6) План “потом”: перенос Supabase на VPS (когда будете готовы)

Коротко: это отдельный большой проект. На практике есть 3 сценария:

1) **Оставить Supabase Cloud навсегда** (часто самый прагматичный путь).
2) **Self-hosted Supabase (полный набор сервисов)** — больше DevOps, но совместимость ближе к “как сейчас”.
3) **Только Postgres на VPS + свой Auth/Storage** — гибко, но это “переписать часть продукта”.

Рекомендация для вашей текущей цели:
- сначала стабилизировать “VPS + микросервисы”
- затем решить, что именно переносить (Auth/DB/Storage), и вернуться к `docs/MIGRATION_PLAN.md`
