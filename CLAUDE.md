# CLAUDE.md — Правила кодирования проекта

Этот файл читается Claude автоматически. Следуй этим правилам во всех сессиях.

---

## Архитектура

- **Frontend:** `src/features/{feature-name}/{components,pages,services,hooks,types,constants}/`
- **Backend:** `ai-consultant-api/src/{routes,services,tools,middleware}/`
- Один файл — одна ответственность. Не смешивать логику и UI.
- Новые фичи — только в `src/features/`, никаких файлов в корне `src/`.

---

## TypeScript

- `strict: true` — всегда. Никаких `any`, кроме крайних случаев с явным комментарием `// TODO: типизировать`.
- Явные возвращаемые типы у функций сервисов и хуков.
- Интерфейсы — без префикса `I`. Плохо: `IEquipment`. Хорошо: `Equipment`.
- `async/await` вместо `.then().catch()`.
- Импорты с расширением `.js` в backend (NodeNext moduleResolution).

---

## Структура файлов backend (ai-consultant-api)

Каждый файл начинается с JSDoc-блока:

```typescript
/**
 * fileName.ts
 *
 * Краткое описание что делает файл.
 *
 * Структура / что умеет:
 *   1. функция_1 — описание
 *   2. функция_2 — описание
 *
 * Пример использования (если уместно):
 *   input → output
 */
```

Секции внутри файла разделяются комментарием:
```typescript
// ============================================
// Название секции
// ============================================
```

## Структура файлов frontend (src/)

- JSDoc на публичных интерфейсах и типах.
- Комментарии только там где логика неочевидна.
- CSS — отдельный файл рядом с компонентом (`Component.css`), не inline styles.

## Именование

| Что | Стиль | Пример |
|-----|-------|--------|
| Компоненты React | PascalCase | `ChatWidget.tsx` |
| Хуки | camelCase с `use` | `useChat.ts` |
| Сервисы | camelCase | `consultantApi.ts` |
| Утилиты | camelCase | `dateFormatting.ts` |
| Типы/интерфейсы | PascalCase | `ParsedInvoice` |
| Константы | UPPER_SNAKE | `MAX_AGENT_ITERATIONS` |
| SQL таблицы | snake_case | `water_invoices` |
| ENV переменные frontend | `VITE_` префикс | `VITE_AI_CONSULTANT_API_URL` |

## Git коммиты

Формат: `тип(scope): описание на русском`

Типы: `feat`, `fix`, `docs`, `refactor`, `chore`

Примеры:
```
feat(invoices): добавить детализацию по адресам
fix(parser): убрать разделитель тысяч из чисел
refactor(water): вынести хук useDeviceArchive
docs(readme): обновить переменные Railway
```

## Supabase

- Все миграции — в `supabase/migrations/` с именем `YYYYMMDD_название.sql`.
- Один файл миграции на одну фичу. При изменении существующей таблицы — обновлять тот же файл (не создавать кусочные патчи).
- Всегда добавлять `IF NOT EXISTS` / `IF EXISTS` для безопасного повторного запуска.
- RLS — обязательно для каждой новой таблицы.
- `service_role` обходит RLS — использовать только в backend.

---

## AI инструменты (tools)

- Каждый инструмент: понятное `description` на русском — агент читает его при выборе инструмента.
- Описание должно говорить **когда использовать** и **когда НЕ использовать**.
- Не писать в description "ОБЯЗАТЕЛЬНО вызывай" — это провоцирует лишние вызовы.

---

## Что НЕ делать

- Не создавать вспомогательные файлы-утилиты для одноразовых операций.
- Не добавлять комментарии к коду который не менялся.
- Не рефакторить код вокруг исправляемого бага.
- Не добавлять обработку ошибок для ситуаций которые не могут возникнуть.
- Не создавать `.md` файлы если пользователь не просил.
- Не пушить в GitHub без явного запроса пользователя.
